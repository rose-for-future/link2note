# 评测 / 发版前能力回归

**纪律：每次上传 GitHub / 发版前，必须跑下面两步，全绿才能发。** 防止改代码或清理时把已有能力改坏。

## 两步回归

```bash
# 1) 单元测试（代码逻辑：router/render/config/models/process/github）
.venv/bin/python -m pytest -q          # 期望：33 passed, 4 deselected

# 2) 能力回归（抓取层：真实链接重跑，对比 baseline）
.venv311/bin/python evals/regression.py   # 期望：4/4 通过，退出码 0
```

两步都过 → 可发版。`regression.py` 退出码非 0 表示有能力缺失，**不准发**。

## 设计

- **`baseline.json`**：每个平台一条"已知良好"的抓取结果（抖音/播客/小红书/GitHub）。
- **`regression.py`**：重跑各平台**抓取层**，与基线对比。
  - 只测抓取（快，且平台改版最容易在这里崩），**不跑耗时的完整转写**。
  - **硬检查（能力）**：抓取成功 + 关键字段非空（标题/音频可解析/图片数/README）。任一失败 → 退出码 1。
  - **软检查（漂移）**：标题/作者/图片数与基线的差异，只告警不判失败（内容可能被作者改）。

## 维护（评测集是攒出来的）

- 每发现一条抓失败的链接、或一个新平台，就**加进 `baseline.json`** 成为新的回归样本。
- 平台改版导致某条 FAIL，先修抓取，再更新对应基线。
- 后续可扩展：转写准确率（CER vs 参考稿）、总结质量（LLM-as-judge 按 rubric 打分）——见设计讨论，先把抓取层这道闸守住。
